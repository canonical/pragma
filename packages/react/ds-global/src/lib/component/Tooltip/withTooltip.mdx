# withTooltip

`withTooltip` is a higher-order component: give it a component type and a message, and it returns a new component that carries a hover tooltip wherever it is rendered. It is the HOC form of the tooltip primitive — sugar over [`TooltipArea`](?path=/docs/components-tooltip-tooltiparea--docs), which internally drives `useDisclosure` in `hover` mode.

```tsx
import { withTooltip, Button } from "@canonical/react-ds-global";

const TooltippedButton = withTooltip(Button, <span>Tooltip content</span>);

// Use it exactly like the component you wrapped:
<TooltippedButton>Hover me</TooltippedButton>;
```

<Canvas of={Stories.Default} />

The tooltip opens when the pointer enters the wrapped element or when it receives keyboard focus, and closes when the pointer leaves or focus moves away, each after a short delay. It is **hover-mode only** — see [It is hover-only](#it-is-hover-only--you-cannot-make-it-open-on-click) below.

## When to reach for it

There are three ways to attach a tooltip in this library, and they differ by _what you have to hand_:

Reach for `withTooltip` when the tooltip belongs to the component itself — the same message, every time it is rendered — and you would rather define it once at module scope than repeat a wrapper at each call site. Because it returns an ordinary component, the tooltipped version composes anywhere the original did.

Internally the HOC renders `<TooltipArea Message={Message} {...popupProps}><Component … /></TooltipArea>`, so `withTooltip` is nothing more than `TooltipArea` with a component type curried in. When you instead have a **one-off JSX node** to annotate — a specific `<span>`, an inline element that only needs a tooltip in this one place — reach for [`TooltipArea`](?path=/docs/components-tooltip-tooltiparea--docs) and wrap the fragment directly. The two are interchangeable in behaviour; the choice is purely ergonomic. The [`Tooltip`](?path=/docs/components-tooltip--docs) component itself is the presentational message bubble and is rarely used on its own.

## Basic example

Define the tooltipped component at module scope — not inside render, so its identity is stable across renders — then use it like any other component:

```tsx
import { withTooltip, Button } from "@canonical/react-ds-global";

const TooltippedButton = withTooltip(Button, <span>Tooltip content</span>);

function Toolbar() {
  return <TooltippedButton>Hover me</TooltippedButton>;
}
```

The `Message` can be any `ReactNode` — a plain string, a `<span>`, or richer markup with an icon.

The returned component's `displayName` is set to `` `withTooltip(${Component.displayName || Component.name || "Component"})` `` — so `withTooltip(Button, …)` appears in React DevTools as **`withTooltip(Button)`**. The inner label resolves in order: the wrapped component's own `displayName`, then its function `name`, falling back to the literal `"Component"` when neither is available (for example, an anonymous arrow function).

## Placement and options

Force the tooltip to prefer one side. Placements are **logical** — `inline-*`
follows the reading direction (so `inline-end` is the right edge in LTR, the left
in RTL) and `block-*` is the vertical axis (`block-start` = top, `block-end` =
bottom). Here it opens to the reading-flow end, falling back only if that edge
does not fit:

```tsx
const InfoButton = withTooltip(Button, "Opens in a new tab", {
  preferredDirections: ["inline-end", "block-start", "block-end", "inline-start"],
});
```

To pin placement to a single side, pass a one-element array:

```tsx
const TopOnly = withTooltip(Button, "Always above", {
  preferredDirections: ["block-start"],
});
```

Push the tooltip further from its anchor, cap a long message's width, and keep it clear of the viewport edge when it appears near a boundary:

```tsx
const RoomyTooltip = withTooltip(Button, longExplanation, {
  distance: "12px",
  maxWidth: "24rem",
  autoFit: true,
  gutter: "8px",
});
```

With `autoFit` enabled, the tooltip is nudged back into view when no preferred direction fits cleanly, and `gutter` keeps a margin between the bubble and the viewport edge while it fits.

Delay opening so a fast pointer sweep does not flash tooltips, and hold the tooltip open briefly after the pointer leaves:

```tsx
const DeliberateTooltip = withTooltip(Button, "Delete permanently", {
  activateDelay: 400,
  deactivateDelay: 100,
});
```

## Pitfalls and behaviour

This primitive has a few sharp edges worth understanding before you rely on it.

### It is hover-only — you cannot make it open on click

`withTooltip` wraps `TooltipArea`, which hard-codes `useDisclosure({ mode: "hover" })`. That is precisely why `popupProps` is typed `Omit<TooltipAreaProps, "children" | "Message">` — and `TooltipAreaProps` itself already drops `mode` (`extends Omit<UseDisclosureProps, "mode">`), so `mode` never reaches the surface and cannot be passed at all. The overlay opens on pointer-enter and focus, and closes on pointer-leave and blur, each after its delay. There is **no way** to make `withTooltip` toggle on click. If you need a click-triggered overlay — a menu, a popover, a dropdown — that is not a tooltip; use the click-mode primitives built on the same `useDisclosure` hook instead.

### Define the wrapped component outside render

`withTooltip` returns a new component. Call it at module scope (or memoise it), not inline in a render body:

```tsx
// Good — stable identity across renders.
const Saved = withTooltip(Button, "All changes saved");

function Bar() {
  return <Saved>Save</Saved>;
}
```

```tsx
// Avoid — a new component type every render remounts the subtree.
function Bar() {
  const Saved = withTooltip(Button, "All changes saved"); // don't
  return <Saved>Save</Saved>;
}
```

Because `Message` and `popupProps` are captured when you call `withTooltip`, they are fixed for that wrapped component. A message that must change at runtime is a sign you want [`TooltipArea`](?path=/docs/components-tooltip-tooltiparea--docs) — where `Message` is a live prop — rather than the HOC.

### The ref does not reach the wrapped component

Internally the HOC renders `<Component {...props} />` — a **plain function component**, not wrapped in `forwardRef`. So a `ref` you put on the returned component is **not forwarded** to the wrapped `Component`; React consumes it at the HOC boundary and it goes nowhere. (This is by design — the tooltip's positioning anchors on `TooltipArea`'s own wrapper, not the wrapped component, so nothing here needs the ref.)

Crucially, **this does not break the tooltip.** Positioning does not anchor to the wrapped component at all. `TooltipArea` renders its own wrapper `<span class="target">` around your component, carrying `aria-describedby` and the measuring ref, and the bubble is placed relative to _that_ span's bounding box. So a component that never sees a ref still gets a correctly-positioned tooltip.

The gap only bites if _your_ component needs a ref for its own reasons — imperative focus, measurement, integration with another library. `withTooltip` cannot supply that; you would need `TooltipArea` (or your own `forwardRef` wrapper) with a component that forwards its ref.

### SSR renders nothing but the trigger

`TooltipArea` renders the tooltip message through `createPortal` into `document.body` (or a `parentElement` you supply). The portal is gated on a `useIsMounted` flag, which is `false` during server rendering **and** the first client render, then `true` after mount — so the server HTML and the first client render agree (neither emits the message) and hydration never mismatches. The server therefore renders only the trigger and its `aria-describedby` target wrapper; the message portals in after hydration. In effect the tooltip is dormant until hydration — the trigger renders on the server, the floating bubble is a client-only concern. This keeps the component build-safe in server environments and hydration-clean on the client.

### How positioning actually anchors

The anchor for positioning is `TooltipArea`'s own inner `<span class="target">`, which carries the measuring ref — **not** the component you wrapped. The bubble is placed relative to that span's bounding box, trying your `preferredDirections` in order and, when `autoFit` is enabled, nudging the bubble to stay within the viewport `gutter`. Because the span wraps your component tightly, this is usually exactly what you want; but be aware the anchor box is the wrapper span, so if your component's inline layout differs from what you expect, the tooltip anchors to the span rather than to your component's own border box.

## Accessibility

`TooltipArea` gives the tooltip message a stable id and sets `aria-describedby` on the target wrapper pointing at it, so assistive technology announces the tooltip as a _description_ of the trigger. This is correct for tooltips: the content supplements the trigger rather than labelling it.

Two consequences follow. First, the trigger must be reachable and focusable for keyboard and screen-reader users to ever surface the tooltip — hover alone excludes them. Wrap something focusable (a `<button>`, a link), not inert text. Second, a tooltip is a _description_, not an accessible _name_: do not rely on `withTooltip` to name an icon-only control. Give that control its own `aria-label` and use the tooltip for supplementary detail.
