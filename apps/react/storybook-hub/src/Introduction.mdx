# Pragma Design System

Pragma is Canonical's design system for building consistent, accessible web interfaces. It provides React components, a token-driven styling architecture, and tooling that bridges design decisions to working code.

## Quick start

```bash
bun add @canonical/react-ds-global @canonical/styles @canonical/ds-assets
```

```tsx
import "@canonical/styles/fonts";
import "@canonical/styles";
import { Button } from "@canonical/react-ds-global";

function App() {
  return (
    <Button anticipation="constructive">
      Save changes
    </Button>
  );
}
```

See [Getting Started](?path=/docs/docs-getting-started--docs) for the full installation guide.

## Concepts

The design system rests on a set of interlocking architectural ideas. Understanding a few of them early will make the rest of the documentation more readable:

**Modifier families** — Components express semantic intent through orthogonal modifier axes (importance, anticipation, criticality, emphasis, lifecycle, release). A single ancestor CSS class shifts the visual output of every descendant without prop drilling. See [Modifiers](?path=/docs/docs-modifiers--docs).

**Token architecture** — Colors flow through a three-tier system (primitives, semantic tokens, channel variables) that handles light/dark theming, modifier contexts, and surface depth through the CSS cascade. Components never hardcode color values.

**Surface depth** — Nested `.surface` containers create visual layers (card on page, popover on card) using compound descendant selectors. No JavaScript, no depth props. See [Surfaces & Themes](?path=/docs/docs-surfaces--docs).

**Subgrid layout** — The grid system and form layout use CSS subgrid for column inheritance, enabling flexible compositions without rigid containers. See [Grid](?path=/docs/docs-grid--docs).

**Ontology-driven specifications** — Components implement specifications from a semantic knowledge graph (the Design System Ontology). Each component has a formal anatomy, modifier bindings, and token contracts defined in the ontology before implementation begins.

**Tiered package architecture** — Packages are organised in tiers (global → global-form → app → app-specific). Each tier extends the one above. Higher tiers may re-export lower tiers so consumers import from a single entry point.

**Code standards** — A standards system (`@canonical/code-standards`) encodes conventions for CSS, TypeScript, accessibility, and component structure as queryable rules with concrete dos and don'ts.
